# Supabase 프로젝트 구성 계획

> 이 문서에서는 사용자가 요청한 `supbase`를 공식 서비스 이름인 `Supabase` 기준으로 정리합니다.

## 현재 인증 방식

현재 Supabase 프로젝트는 이메일 로그인만 허용되어 있으므로, 앱도 이메일 회원가입/로그인 방식으로 구성했습니다.

앱 동작 방식은 다음과 같습니다.

1. 로그인하지 않은 사용자는 이메일 로그인/회원가입 화면만 볼 수 있습니다.
2. 로그인에 성공하면 Todo 입력창과 목록이 표시됩니다.
3. Todo 데이터는 로그인한 사용자의 `user_id`로 `todos` 테이블에 저장됩니다.
4. RLS 정책 때문에 사용자는 자기 Todo만 조회, 추가, 수정, 삭제할 수 있습니다.
5. 로그아웃하면 다시 로그인 화면으로 돌아갑니다.

## 목표

현재 Todo 앱은 Supabase Auth로 로그인한 사용자만 사용할 수 있습니다. Todo 데이터는 Supabase PostgreSQL 데이터베이스에 저장되고, 사용자별로 분리됩니다.

현재 앱의 Todo 데이터 형태는 다음과 같습니다.

```js
{
  id: Number,
  text: String,
  priority: "high" | "medium" | "low",
  completed: Boolean
}
```

Supabase에서는 기존 필드를 유지하되, 데이터베이스에 맞게 `id`, `position`, `user_id`, `created_at`, `updated_at`을 추가하는 구성을 권장합니다.

## 가입 전 결정할 것

1. 인증 없이 공용 Todo 앱으로 쓸지, 사용자 로그인 기반 Todo 앱으로 쓸지 결정합니다.
2. 이 앱은 개인 Todo 성격이 강하므로 최종 구조는 `Supabase Auth + RLS`를 사용하는 사용자별 Todo 구성을 권장합니다.
3. 현재 앱에는 이메일 로그인 UI가 추가되어 있으므로 사용자별 Todo 저장 구조를 사용합니다.

| 방식 | 장점 | 단점 | 추천도 |
| --- | --- | --- | --- |
| Supabase Auth 사용 | 사용자별 데이터 분리 가능 | 로그인 UI 추가 필요 | 실제 앱 추천 |

이 문서는 실제 앱으로 확장하기 쉬운 `Supabase Auth 사용` 기준으로 작성합니다.

## Supabase 프로젝트 생성 순서

1. Supabase에 가입합니다.
   - https://supabase.com
2. 새 프로젝트를 생성합니다.
   - Organization: 기본 조직 사용
   - Project name: `todo-app` 또는 원하는 이름
   - Database password: 안전한 비밀번호를 생성해서 따로 보관
   - Region: 한국 사용자가 중심이면 가까운 아시아 리전 선택
3. 프로젝트 생성이 끝나면 다음 값을 확인합니다.
   - Project URL
   - Publishable key 또는 anon public key
4. 프론트엔드에서 사용할 값은 공개 가능한 키만 사용합니다.
   - `service_role` 키는 절대 브라우저 JavaScript에 넣지 않습니다.

## 권장 테이블 구조

테이블 이름은 `todos`를 권장합니다.

| 컬럼 | 타입 | 필수 | 기본값 | 설명 |
| --- | --- | --- | --- | --- |
| `id` | `bigint` | 예 | identity | Todo 고유 ID |
| `user_id` | `uuid` | 예 | `auth.uid()` | Todo 소유 사용자 |
| `text` | `text` | 예 | 없음 | Todo 내용 |
| `priority` | `text` | 예 | `'medium'` | `high`, `medium`, `low` 중 하나 |
| `completed` | `boolean` | 예 | `false` | 완료 여부 |
| `position` | `integer` | 예 | `0` | 드래그 정렬 순서 |
| `created_at` | `timestamptz` | 예 | `now()` | 생성 시간 |
| `updated_at` | `timestamptz` | 예 | `now()` | 수정 시간 |

### 왜 `position`이 필요한가

현재 앱은 배열 순서로 드래그 정렬 상태를 저장합니다. 데이터베이스에는 배열 순서가 없으므로, 사용자가 드래그해서 바꾼 순서를 보존하려면 `position` 컬럼이 필요합니다.

### 왜 `user_id`가 필요한가

브라우저에서 Supabase에 직접 접근하는 구조에서는 사용자별 접근 제어가 중요합니다. `user_id`를 저장하고 Row Level Security 정책에서 `auth.uid()`와 비교하면, 로그인한 사용자는 자기 Todo만 읽고 수정할 수 있습니다.

## SQL 생성 스크립트

Supabase Dashboard의 SQL Editor에서 아래 SQL을 실행합니다.

```sql
create table public.todos (
  id bigint generated by default as identity primary key,
  user_id uuid not null default auth.uid() references auth.users(id) on delete cascade,
  text text not null check (length(trim(text)) > 0),
  priority text not null default 'medium' check (priority in ('high', 'medium', 'low')),
  completed boolean not null default false,
  position integer not null default 0,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

create index todos_user_id_position_idx
  on public.todos (user_id, position);

alter table public.todos enable row level security;
```

이미 `todos` 테이블을 만들었다면 위 SQL을 다시 실행하지 말고, Supabase Dashboard의 Table Editor에서 컬럼이 이 문서와 같은지 확인합니다.

이미 만든 테이블에서 `user_id` 기본값이 비어 있다면 아래 SQL을 실행합니다.

```sql
alter table public.todos
alter column user_id set default auth.uid();
```

## updated_at 자동 갱신

Todo를 수정할 때마다 `updated_at`이 자동으로 바뀌도록 트리거를 추가합니다.

```sql
create or replace function public.set_updated_at()
returns trigger
language plpgsql
as $$
begin
  new.updated_at = now();
  return new;
end;
$$;

create trigger set_todos_updated_at
before update on public.todos
for each row
execute function public.set_updated_at();
```

## RLS 정책

Supabase에서 브라우저 앱은 공개 가능한 API 키를 사용합니다. 따라서 테이블에는 RLS를 켜고, 로그인한 사용자가 자기 데이터만 접근하도록 정책을 만들어야 합니다.

```sql
create policy "Users can read own todos"
on public.todos
for select
to authenticated
using ((select auth.uid()) = user_id);

create policy "Users can create own todos"
on public.todos
for insert
to authenticated
with check ((select auth.uid()) = user_id);

create policy "Users can update own todos"
on public.todos
for update
to authenticated
using ((select auth.uid()) = user_id)
with check ((select auth.uid()) = user_id);

create policy "Users can delete own todos"
on public.todos
for delete
to authenticated
using ((select auth.uid()) = user_id);
```

## 앱에서 사용할 데이터 형태

Supabase에서 읽은 Todo는 다음 형태로 다루면 됩니다.

```js
{
  id: 1,
  user_id: "사용자 uuid",
  text: "할 일",
  priority: "medium",
  completed: false,
  position: 0,
  created_at: "2026-05-20T00:00:00.000Z",
  updated_at: "2026-05-20T00:00:00.000Z"
}
```

기존 화면 렌더링에는 `id`, `text`, `priority`, `completed`만 필요합니다. 드래그 정렬 저장에는 `position`을 추가로 사용합니다.

## 이메일 로그인 설정

현재 앱은 아래 Supabase Auth 메서드를 사용합니다.

```js
supabaseClient.auth.signUp({ email, password });
supabaseClient.auth.signInWithPassword({ email, password });
supabaseClient.auth.signOut();
```

Supabase Dashboard에서 아래 설정을 확인합니다.

1. Supabase Dashboard에서 현재 프로젝트를 엽니다.
2. 왼쪽 메뉴에서 `Authentication`으로 이동합니다.
3. `Providers` 또는 `Sign In / Providers` 메뉴를 엽니다.
4. `Email` Provider가 활성화되어 있는지 확인합니다.
5. 이메일 인증 메일을 사용할 경우, 회원가입 후 받은 메일에서 인증을 완료해야 로그인할 수 있습니다.
6. 이메일 인증 링크가 로컬 앱으로 돌아오게 하려면 `Authentication > URL Configuration`의 Site URL을 개발 중에는 `http://localhost:5500`으로 설정합니다.

2. Authentication 설정에서 Email Provider가 활성화되어 있는지 확인합니다.
3. 앱을 `http://localhost:5500`으로 엽니다.
4. 이메일과 비밀번호로 회원가입합니다.
5. 이메일 인증이 켜져 있다면 인증 메일을 확인합니다.
6. 로그인 후 할 일을 하나 추가합니다.
7. Supabase Dashboard의 Table Editor에서 `todos` 테이블에 row가 생겼는지 확인합니다.
8. row가 생기면 Supabase 저장이 정상 동작하는 상태입니다.

## 주의할 점

- 브라우저에는 `Project URL`과 공개 가능한 API 키만 넣습니다.
- `service_role` 키는 서버 전용이므로 절대 노출하지 않습니다.
- RLS를 켠 뒤 정책이 없으면 브라우저에서 데이터가 조회되지 않는 것이 정상입니다.
- `priority`는 DB check 제약으로 `high`, `medium`, `low`만 허용합니다.
- 빈 Todo는 현재 앱에서도 막고, DB에서도 `check (length(trim(text)) > 0)`로 한 번 더 막습니다.
- 드래그 정렬은 단일 row 수정이 아니라 여러 row의 `position` 업데이트가 필요할 수 있습니다.
